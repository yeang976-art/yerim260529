# Backend Practice Archive

## Overview

이 문서는 Java/Spring 백엔드 부트캠프 과정에서 진행한 실습과 과제 기록을 정리하기 위한 아카이브입니다.

실습을 단순히 제출하고 끝내는 것이 아니라, 어떤 문제를 만나고 어떻게 이해했는지 기록해 백엔드 개발 포트폴리오로 발전시키는 것이 목적입니다.

## Practice 01. API Spec & Postman Mock Server

### Summary

맛집 관리 API를 기준으로 API 명세서를 작성하고, Postman을 활용해 Mock Server 응답을 구성했습니다.

### What I Practiced

- API Endpoint 설계
- HTTP Method 구분
- Request Body 작성
- Response 예시 작성
- Status Code 정리
- Postman Collection 구성
- Mock Server 응답 확인

### What I Learned

API 명세서는 프론트엔드와 백엔드가 같은 기준으로 개발하기 위한 약속이라는 점을 이해했습니다.

또한 Mock Server는 실제 비즈니스 로직을 처리하는 서버가 아니라, 미리 정의한 Example 응답을 바탕으로 API 동작을 가정해볼 수 있는 가상 서버라는 점을 알게 되었습니다.

## Practice 02. DevTools API Analysis

### Summary

Chrome DevTools Network 탭을 활용해 웹 페이지에서 발생하는 API 요청을 분석했습니다.

### What I Practiced

- Network 탭 확인
- Fetch 필터 사용
- Request Method 확인
- Header 확인
- Status Code 확인
- Response Body 확인
- OPTIONS 요청과 GET 요청 구분

### What I Learned

처음에는 Others 필터에서 확인한 OPTIONS 요청을 실제 데이터 조회 요청으로 착각했습니다.

하지만 OPTIONS 요청은 실제 데이터를 가져오는 요청이 아니라, 서버와의 연결 가능 여부와 CORS 허용 정책을 확인하는 사전 요청이라는 점을 알게 되었습니다.

과제에서 요구한 것은 Fetch 필터에서 확인할 수 있는 GET 요청 분석이었고, Response 탭에서 JSON 데이터를 확인하려면 실제 GET 요청을 봐야 한다는 점을 이해했습니다.

같은 URL이라도 Method에 따라 역할이 달라질 수 있으므로, API를 분석할 때는 URL뿐 아니라 Method, Status Code, Header, Response Body를 함께 확인해야 합니다.

## Practice 03. Git & GitHub Workflow

### Summary

Git과 GitHub를 활용해 브랜치를 나누고, Markdown 파일을 작성한 뒤 Pull Request를 통해 변경 사항을 병합하는 흐름을 연습했습니다.

### What I Practiced

- Branch 생성
- Markdown 파일 작성
- Commit 작성
- Pull Request 생성
- Self Review 작성
- Merge 진행
- `.gitignore` 설정

### What I Learned

Git은 단순한 파일 백업 도구가 아니라 변경 이력을 기록하고 관리하는 버전 관리 도구입니다.

GitHub는 Git으로 관리한 기록을 원격 저장소에 올리고, Pull Request를 통해 변경 내용을 리뷰하고 병합할 수 있게 해주는 협업 플랫폼입니다.

또한 IntelliJ에서 생성되는 `.idea` 같은 개인 설정 파일은 협업 저장소에 포함하지 않는 편이 좋으며, `.gitignore`를 통해 불필요한 파일 추적을 막을 수 있다는 점을 배웠습니다.
